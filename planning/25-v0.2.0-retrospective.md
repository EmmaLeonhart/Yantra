## The GUI, precisely ‚Äî what runs where

The host/substrate split holds, and the host does **not** recompute any
displayed value:

- **Substrate (decoded by the host, not recomputed):**
  - `frame.su` ‚Äî `pixel(x,y) = make_real(1 - x*x - y*y)`. The field.
  - `toggle.su` ‚Äî `flip(s) = make_real(1 - s)`. The colour-state transition.
  - The host reads both only via `vsa.real(...)` and never computes
    `1 - x*x - y*y` or `1 - s` itself (grep-confirmed).
- **Host (I/O only, as declared):** grid assembly, the tint/colormap
  (numpy), the tkinter window, click delivery.

**Precision nuance (sharpening, not a correction):** the arithmetic in
`pixel`/`flip` is **scalar** arithmetic that Sutra compiles to torch ops
on the device, with `make_real` lifting the final scalar onto the VSA real
axis; the host decodes with `real()`. This is genuine substrate execution
(Sutra-compiled, device-resident, decoded from the substrate) and is the
**same model the calculator uses** ‚Äî the calc's actual `+ ‚àí √ó √∑` are also
runtime arithmetic; only the operator *select* lives in VSA space. It is
*not* arithmetic performed inside the high-dimensional bundled/rotation-
bound VSA representation. "Computed on the substrate" is accurate and
consistent with the calc; it would be an overclaim only if read as
"computed inside the VSA bundle," which neither the GUI nor the calc
claims.

## Verdict on the "foundation of the OS" framing

**Defensible, because it is scope-limited.** Yantra now puts
substrate-computed visual output on a screen that mutates at runtime via
substrate compute ‚Äî a genuine first, and the v0.2.0 notes state plainly
everything it is *not*:

- The window is **host tkinter** ‚Äî a stand-in for the unbuilt Yantra-native
  / browser GUI (build-sequence milestone 3).
- Render batching is **blocked** on a Sutra-side `make_real`-batch change;
  per-pixel render is slow.
- The **reverse-CNN decoder** (return one vector ‚Üí reorganise into a frame,
  Emma's fuller form) is **unbuilt** ‚Äî the bigger next GUI step
  (`planning/24`).
- The OS foundations named in CLAUDE.md remain **unbuilt**: the bootloader,
  the Rust orchestrator, disc‚ÜîRAM‚ÜîGPU storage-tier moves, the
  multi-threaded Sutra runtime, boot-time hardware discovery.

So: a **proof-of-concept foundation**, not a finished OS GUI. Read with
those limits, "foundation of the OS" holds; read as "the OS GUI works," it
would not ‚Äî and the release notes already prevent that misreading.
