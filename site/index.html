Yantra — early development

The OS where the symbol is the computation.

Yantra is a neuro-symbolic, GPU-native operating system written in Sutra. Kernel, processes, IPC, and GUI are not programs that run on a substrate — together they are one differentiable tensor-op graph.

The CPU shrinks to a sequencer that boots the machine and shuffles inactive work between GPU and RAM. Everything that matters happens as Sutra programs exchanging axons — structured embeddings produced by rotation binding — so local AI plugs in with no translation layer between thought and action.

Why it exists

Today's stacks were fitted around scarce serial compute and disembodied language models. As GPUs become the default substrate and AI systems want to reason continuously rather than emit strings, the same impedance mismatch shows up everywhere: models think in vectors, but software speaks in bytes. Every round trip pays a translation tax.

Today model thought → text → parse → execute → output → re-embed → model thought On Yantra model activation → Sutra program → tensor output → directly consumable

Because every Sutra program already lives in the embedding space the model is thinking in, perception, reasoning, and action are first-class operations on one representation. There is no translation layer because there is nothing to translate to. This is the substrate beneath tool-use and agents — not an LLM that calls tools.

The architecture

The kernel is a Connectome Manager, not a traditional scheduler. Its job is deciding what is connected to what — moving programs between three tiers rather than context-switching them.

Axons

The universal data type. Processes exchange structured embeddings bound by rotation, so a capability is just a role and IPC is just vector algebra — no serialization boundary.

Three tiers

Disc holds programs at rest, RAM holds them loaded but idle, GPU holds the live connectome. The kernel decides which programs are wired in right now.

CPU as orchestrator

A small Rust orchestrator boots the system and routes axons. The heavy computation is Sutra running on the GPU — the CPU never pretends to be the runtime.

The build sequence

Deliberately ordered. Each phase ships before the next begins.

Connectome Manager

The runtime that decides what runs on the GPU vs. sits in RAM vs. on disc. A tested v0.0 nucleus exists in Sutra + a Python orchestration shim; the production form is Rust.

Command-line userspace

Linux-shaped file utilities (cat, ls, grep…) written natively in Sutra. First access is command-line only, over SSH or serial from a host. No GUI in this phase.

Browser & GUI

Only after the first two ship. Every UI surface — start menu, cursor, login, file manager — is a browser-rendered HTML page. One framework: HTML5 + CSS + TypeScript + WebGL, no WASM.

What's real, and what isn't

This repository is mostly planning, plus a working v0.0 kernel nucleus. Naming the gaps plainly matters more than the pitch:

Working today

  • Sutra v0.6.0 — the typed, differentiable language, compiler, and runtime Yantra is built in.
  • v0.0 kernel nucleus — compiles .su services and routes axons producer→…→sink through a capability-checked router.
  • CLI calculator — type an expression, exact + − × ÷ computed on the substrate; verified exact or refused (52 tests). See the demo below.
  • Multi-process Sutra runtime — shared-state services running through the router.
  • Disc↔GPU load/unload — GPU memory genuinely reclaimed, measured on a real RTX 4070.
  • TypeScript→Sutra — lowering engine + ts2su CLI, 17 fixtures passing.

Not built yet

  • The bootloader — bare-metal CPU + GPU init. Does not exist.
  • The Rust orchestrator — the Python prototype is its behavioural reference, not a smaller version of it.
  • Per-process GPU memory arenas + GPU-tick-parallel scheduling.
  • State-preserving eviction to RAM — needs a Sutra serialise-process-state primitive.
  • The browser layer — HTML/CSS layout engine and WebGL bindings.
  • MMIO / interrupt path and the C→Sutra transpiler.

Yantra targets critical systems — defense, aerospace, industrial, medical, autonomous — where predictable latency, formal verifiability, and a small attack surface matter more than mass-market compatibility. It is not a consumer desktop, and the hard problems above are real, named, and tracked rather than waved past.

See it compute

The smallest definitive version of the bet: a CLI calculator. You type an expression and it prints the answer — computed on the Sutra substrate through the kernel, verified exact. Where a video model would generate a plausible-looking terminal frame (and the number can drift), Yantra computes it — and refuses rather than guess when it can't be exact.

$ python apps/calc/calc.py 5 * 10 = 50 2 + 3 * 4 = 14 (10 - 2) * 5 = 40 10 / 2 = 5 10 / 3 = (refused — not an exact quotient; never a wrong answer)

Full expressions with precedence and parentheses; every result is checked against an exact oracle and refused if it can't be confirmed — 52 tests, all exact-or-refused. The same property at scale: a fidelity harness pushes 1024 distinct symbols through the kernel and recovers every one bit-exact, with zero drift — precisely the axis Meta's own Neural Computers paper lists as unsolved (symbolic stability).
